# ハッカソン企画「パチンコホール群像のLLM心理シミュレーション」— 進め方と方向性の決定相談

## 0. この相談のゴール

実装計画を1つ提示するのではなく、**私がこれから進める方向性を決めるための選択肢・限界・判断軸を提示してほしい**。具体的には:

1. 私の希望（100 ペルソナ・3 機種・群像劇）が実際にどこまで可能かのリアリティチェック
2. 「ここまでなら容易」「これ以上は厳しい」という具体的ライン
3. 3 つ程度の方向性の選択肢（それぞれの成果物イメージ付き）
4. 推奨路線 + 私が判断するための軸
5. ローカル LLM 初心者の私が試行錯誤モードに入るまでの最短ルート
6. 方向性決定後の 3 週間実行プラン

「最適解を1つ押しつける」のではなく、**「私がこの 3 週間をどう使うか決められる材料を揃える」**ことを目指してほしい。

## 1. ハッカソンの提出要件（絶対制約）

「シンギュラボ」コミュニティのハッカソン（締切 **2026-05-07**、残り約3週間）。

主催側から `2d-multi-places-simulation-on-fire-public` というベースコードが配布されている。
これを改造または拡張して成果物にする。

### 提出物（3点）
1. ソースコード（GitHub が理想）
2. **シミュレーション結果の可視化動画**
3. 簡単な解説文章

- デフォルトは 20 エージェント × 100 ステップ × 2 推論/ステップ = 約 4000 LLM call

- 可視化ツール（viewer.html、generate_video.py）も同梱

- 「ollama や python に慣れていない人には難しい」「ある程度のスペックの PC 必須」
- 「Claude / ChatGPT に読み込ませて相談するのを推奨」
- 「参加者同士のコラボも推奨」
- **評価軸は明示されていない**。提出物3点の完成度＋企画の独自性が鍵と推測される

## 2. 企画の全体像

### 最優先の観察目的
1. **最も興奮する瞬間の観測**（ドーパミン相当 proxy のピーク条件の特定）
2. **最も不幸になる瞬間の観測**（絶望・後悔・loss chasing のピーク）
3. 自分が知らない社会的背景の人の内面を知る（共感的動機）
4. **AI が人間のような感情や行動をどう示すかを観察して遊ぶ**（探索的目的）

### 独自仮説（企画の核心）
- ユーザーは 15 年間ギャンブル依存症だった実体験者
- 体感仮説: **「最大ストレス × 大逆転認知の瞬間」に最強のドーパミンが出る**
- 強化版: **「2010年代の爆裂 ART 機」は「2000年代の爆裂 AT 機」より中毒性の質が違う**
  （2010s: 一度連チャンに入ると半日〜1日止まらない／2000s: 出ても1-2時間で終わる）
  - この**世代差の機序をエージェント挙動の差分として観察する**ことが企画の目玉

### 再現したい要素

**(A) 機種パターン 3世代**

| 機種 | 世代 | 特徴 |
|------|------|------|
| 爆裂 AT 機（ミリオンゴッド系） | 2000年代 | 一撃万枚、分散極大、短時間型 |
| 4.5号機（北斗の拳系） | 2000年代後半 | AT/ART 性能高い、波大 |
| 爆裂 ART 機 | 2010年代 | **連チャンが長時間継続（中毒性の本体）** |

**(B) ペルソナ: 30〜100 人規模の群像**

初期案は 5 ペルソナ。ユーザーの強い希望で **30〜100 人規模の多様な群像**を並置し、
**自分が知るリアルなパチンコホールの空間を再現したい**。
「色々な人間に色々な感情が起こる様を観測したい」「AI が人間っぽくどう振る舞うか見たい」
という**探索的動機**が核心。

中核の 5 人:
| # | ペルソナ | 観察視点 |

| a | 借金を抱えた学生 | 返済圧 × 若さの無計画 |

年齢（20代〜70代）、性別、職業、依存度、所持金、来店動機、心理状態などで分散。

**(C) 観察アウトカム**
- 覚醒語彙の自然発生頻度（「まだいける」「熱い」「来る」「取り返せる」等）
- 不幸語彙の頻度（「終わった」「絶望」「後悔」等）

- ペルソナ間の感情表現の質的差
- 機種間の継続ステップ数の差
- 群像全体として「ホールの熱量」のピーク

## 4. すでに確定した技術前提

前回の ChatGPT Pro 相談（DR の前の初回相談）で以下が確定:

- 「巨大モデル一点賭け」ではなく「中型 × non-thinking × 短文出力 × 最後だけ上位」戦略
- `message` と `action` を別モデルに分けるのが有効
- 仮説精密化: 「stress 単独」ではなく「**stress × comeback-cue の相互作用**」
- 撤退ゲート: **4/21 stock demo 動作 / 4/25 pilot 通過 / 4/29 2×2 ログ / 5/02 図+主張**
- 縮退順序: action 捨てる → message-only → ルールベース + LLM 解釈のみ
- Ollama 罠: thinking 切り忘れ / context 積みすぎ / keep_alive / num_predict / GPU fallback / server overload
- 「定量 only ≠ 無名ベクトル」（`stress_index=0.83` のように名前付きで渡す）

# パチンコホール群像 LLM シミュレーションの最適実行構成

## 結論

**事実と出典** 2026年4月時点で、Ollama には Qwen3 系の成熟した公式 tag 群があり、`qwen3:4b` は 2.5GB、`qwen3:14b` は 9.3GB、`qwen3:30b` は 19GB で公開されています。Qwen3.5 系も公式に `qwen3.5:9b` と `qwen3.5:35b-a3b` があり、後者は Q4_K_M で 24GB です。さらに 2026年3月30日には Apple Silicon 向け MLX プレビューが Qwen3.5-35B-A3B 向けに公式発表されましたが、4月には MLX runner 失敗、NVFP4 系の回帰、`think=false` と structured output の相性問題が継続報告されています。

**推奨と理由** 私の推奨は、**前景 12 人を `qwen3:30b`、背景 48 人を `qwen3:4b`、最終動画を書き出す 1 seed だけ前景 12 人を `qwen3.5:35b-a3b` に差し替える、合計 60 人・48 step・1 call/agent-step の階層構成**です。これが最適な理由は三つあります。第一に、`qwen3:30b` は公式 Ollama tag として十分にこなれており、Qwen3 自身が role-playing・multi-turn dialogue・agent capability を強く打ち出しているため、内面言語を作る主力に向きます。第二に、Qwen3.5-35B-A3B は品質面では魅力的でも、2026年4月の Ollama 上では「速くなったが、まだ boring ではない」状態なので、**全 pilot の backbone**ではなく**最終出力だけの前景強化**に回す方が締切耐性が高いです。第三に、この構成なら 100 人の群像感を**100 ペルソナの設定プール**で確保しつつ、実行時は 60 人に制御して動画の密度と実装安定性を両立できます。

## ローカルLLM事情

**事実と出典** Qwen 系の最新動向はかなり速く、Qwen3 は `4b / 8b / 14b / 30b / 32b / 235b`、Qwen3.5 は `0.8b / 2b / 4b / 9b / 27b / 35b / 122b` が Ollama 公式ライブラリに並んでいます。さらに `qwen3.6` が 2026年4月18日時点で公開直後で、`35b-a3b` 系の新 tag も出ていますが、公開から数時間〜数日の段階です。締切が 2026年5月7日であることを考えると、**性能の魅力**より**成熟度の低さ**が支配的です。

**事実と出典** Apple Silicon 上のランタイム比較では、公式の Ollama MLX プレビューが Qwen3.5-35B-A3B を題材に大幅な高速化を示し、M5 系で decode 112 tok/s、int4 では 134 tok/s を報告しています。一方、Apple Silicon 向けの学術・OSS 側では、vLLM-MLX 系が M4 Max 実験で llama.cpp より 21%〜87% 高い throughput を示し、Qwen3-30B-A3B では 109.7 tok/s、16 concurrent で 4.3 倍の aggregate throughput を報告しています。ただし、この論文の M4 Max は 128GB 機で、**M4 Max 64GB の Ollama 単独・公式条件での厳密な tok/s 公開値は不明**です。現時点の 64GB Mac 向け数値はコミュニティ計測が中心で、Qwen3.5-35B-A3B の実効速度はランタイムと量子化で大きくブレます。

**事実と出典** 代替候補を切る理由もかなり明確です。`llama4:scout` は Ollama 上で 67GB で、公式の supported languages に日本語が入っていません。Meta の Llama 4 を日本語ロールプレイに使うのは、64GB 機でも規模・言語の両面で筋が悪いです。Gemma 3 は 140 以上の言語対応で軽量ですが、Gemma 4 は 2026年4月公開直後で、thinking mode 可変という利点の反面、`think=false` で `format` が silently ignored される問題、OpenAI 互換 endpoint で content が空になる問題、長い system prompt で空応答になる問題が報告されています。Google DeepMind 由来の Gemma 4 を今回の backbone にするのは危険です。DeepSeek-R1 distill は依然として reasoning-first の family で、あなたが既に固定した「non-thinking × 短文出力 × 最後だけ上位」戦略と噛み合いません。

**事実と出典** 日本語特化・日本語強化では、Qwen3-Swallow が `8B / 30B-A3B / 32B` の bilingual Japanese-English family として公開されており、Institute of Science Tokyo と AIST の系統で出ています。ELYZA は 2026年1月に日本語拡散言語モデル ELYZA-LLM-Diffusion を発表し、同社の `ELYZA-Shortcut-1.0-Qwen-32B` も公開しています。KARAKURI も日本語 LLM の製品展開を続けています。ただし、**stock Ollama でそのまま差し込んで 3週間で回す**という観点では、公式ライブラリで即使え、既にユーザー環境で動作確認も済んでいる Qwen 系の方が圧倒的に低リスクです。

**推奨と理由** したがって、今回の「攻めてるが通る」解は、**最新性ではなく成熟度を勝たせる**べきです。つまり、今すぐ backbone に置くべきなのは `qwen3:30b` であり、`qwen3.6` は観察対象、`qwen3.5:35b-a3b` は最終仕上げ専用、Gemma 4 と Llama 4 Scout は採用しない、という判断が最も筋が通ります。

## 多エージェント実現性

**事実と出典** LLM マルチエージェントのスケールそのものは、すでに「25人」から「24〜100人」「1,052人」「10,000人」まで文献上確認できます。Generative Agents は 25 人の Smallville を動かし、2025年の social conventions 研究は 24〜100 体規模で局所相互作用から規範形成を観測し、`Generative Agent Simulations of 1,000 People` は 1,052 人の interview-based agent を構築しました。AgentSociety は 1万体超・500万相互作用を掲げています。つまり、**30〜100人規模の群像シミュレーション自体は、2026年時点では「無謀」ではなく「設計の問題」**です。

**事実と出典** 一方、Apple Silicon 上での serving の現実は別です。Ollama 公式 FAQ では `OLLAMA_NUM_PARALLEL` の default は 1、必要 RAM は `OLLAMA_NUM_PARALLEL × OLLAMA_CONTEXT_LENGTH` に比例して増え、`OLLAMA_MAX_QUEUE` default は 512、`keep_alive` でモデルをメモリ常駐させられると明記されています。加えて、vLLM 本体は Apple Silicon を experimental support と位置づけ、vllm-metal は community-maintained plugin です。vllm-metal には paged attention と vLLM scheduler があり、2026年4月の v0.2.0 で TTFT と throughput 改善をうたっていますが、base code が `localhost:11434` 前提で組まれている以上、Ollama を外すと「性能」は上がっても「統合作業」が増えます。

**推奨と理由** ここから導くべき答えは、**100 ペルソナを作るが、同時稼働は 60 人に止める**です。具体的には、Claude Code に 100 persona pool を生成させ、各 run では固定の前景 12 人とサンプリングされた背景 48 人だけを起動します。これなら「ホールの群像感」は十分に出せますし、しかも 48 step・1 call/agent-step にすれば総 call 数は 2,880 で、配布コードのデフォルト約 4,000 call より軽くなります。さらに Ollama 公式 docs のメモリスケーリングを踏まえると、**global は `OLLAMA_NUM_PARALLEL=2` が安全圏**で、`OLLAMA_MAX_LOADED_MODELS=2`、`OLLAMA_MAX_QUEUE=256`、`OLLAMA_FLASH_ATTENTION=1`、`OLLAMA_KV_CACHE_TYPE=q8_0`、各 API call の `keep_alive=-1` が現実的です。公式には sweet spot は公開されていないので、2 は私の推奨値ですが、64GB の単機で 30B と 4B を同時に安定運用するにはこのくらいが妥当です。

## ペルソナと依存性仮説

**事実と出典** 近年のペルソナ生成・演じ分け研究は、まさにあなたの用途に噛み合っています。Mixture-of-Personas は人口分布に合わせて persona と exemplar を確率的に混ぜる手法で、多様性と alignment を両立しようとします。HAG は hierarchical conditional probability に基づいて macro distribution を合わせつつ micro-level の一貫性を確保する生成枠組みです。CharacterBox は role-playing を静的 QA ではなく「動的な行動軌跡」で評価すべきだと示し、Memory-Driven Role-Playing は長い対話での persona 知識活用を測る MREval / MRPrompt を提案しています。さらに、Anthropic の persona vectors 研究は、character trait の監視や制御を activation-space の方向として扱える可能性を示しました。

**推奨と理由** したがって、ペルソナは手書き 60 本ではなく、**Claude Code に HAG 風の quota 制約を与えて 100 本の数値 persona を一括生成させる**のが正解です。年齢帯、職業、同居家族、可処分現金、借入額、依存重症度、羞恥反応、隔離感、衝動性、損失追跡傾向、会話性、来店動機、機種経験を全部数値または code 化し、runtime にはその numeric state だけを渡します。人間が読むための `caption_ja` と `seed_memory_ja` は別ファイルに残してよいですが、LLM への live input は数値だけにするべきです。この構成なら配布コードの哲学を守りつつ、キャラ崩壊を防ぐ「短い identity refresh」を 8 step ごとに差し込めます。

**事実と出典** ギャンブル側の神経科学も、あなたの仮説をそのまま否定しません。レビュー論文では、不確実で間欠的な報酬が dopamine 系を強く駆動しうること、EGM の structural characteristics がその効果を増幅しうること、near-miss が ventral striatum や dopaminergic midbrain に関連反応を生むこと、loss chasing が gambling disorder の中核症状であることが繰り返し示されています。event frequency や speed of play も、認知制御や継続行動に影響する重要な要素として扱われています。

**推奨と理由** ただし、**「2010年代の爆裂 ART 機は 2000年代 AT 機より中毒性が高い」という日本機種史の head-to-head な査読済み比較は、今回の調査では不明**でした。だから、そこを文献事実として断言するのではなく、**仮説として数値パラメタに落とし込む**のが正しいです。私はこれを `stress × comeback_cue` を中核にした proxy 設計で扱うべきだと考えます。すなわち、2000s AT は `entry_shock` が高く `continuity` が短い、4.5号機はその中間、2010s ART は `continuity_index` と `cue_decay` が大きい、という差で実装し、最終的に「文献が直接証明した」のではなく「構造特性の文献と実体験仮説を合わせた operationalization である」と書くべきです。これなら審査向けの独自性も守れます。

## 推奨実行構成

**モデル構成の推奨** 前景 12 人の message 主力は `qwen3:30b`、背景 48 人の action+short-message は `qwen3:4b`、最終動画 seed の前景 12 人だけ `qwen3.5:35b-a3b` に差し替え。Ollama は `/api/chat` を使い、`think:false`、`stream:false`、`keep_alive:-1`、`OLLAMA_NUM_PARALLEL=2`、`OLLAMA_MAX_LOADED_MODELS=2`、`OLLAMA_MAX_QUEUE=256`、`OLLAMA_FLASH_ATTENTION=1`、`OLLAMA_KV_CACHE_TYPE=q8_0`。

**エージェント配置** 100 persona pool、active 60（foreground 12 固定 + background 48 差し替え）。密観察 12 人はコア 5 人 + 中年自営、年金生活者、回復期ギャンブラー、カップル同行者、ホール常連、高揚しやすい初心者、無感情な観察者。

**場所設計** `AT_bank=16`、`Hokuto_bank=16`、`ART_bank=20`、`entrance`、`walkway`、`smoking`、`rest`、`counter`、`exit` の 9 place 構成。解析用 run は bank-isolated、最終動画は三世代同居ホール。ART 島だけ座席数と近傍 visibility を大きく。

**時間設計** 48 step、1 step = 15 分、1 step ≒ 40〜60 ゲーム相当。`dopamine_proxy = stress_index * comeback_cue + 0.35 * continuity_index * active_run`、`comeback_cue = 0.45*near_miss + 0.25*drop_from_peak + 0.20*neighbor_win + 0.10*signal`。`continuity_index` は `AT=0.35 / 4.5=0.58 / ART=0.82`、`cue_decay_steps` は `1 / 2 / 4`。

**実験セット** 必須 run は 8 本。三世代それぞれ `low stress × low cue` と `high stress × high cue` の 6 本、persona(e) 主役の ART 変容 run を `low stress × high cue` と `high stress × high cue` で 2 本。

**プロンプト骨格** `think:false`、`stream:false`、soft schema + local validator + 1 retry。foreground: `{"action_id":0-5,"target_place_id":0-8,"stake_shift":-1|0|1,"continue_prob":0-100,"arousal":0-100,"despair":0-100,"message":"..."}`、1 文 24 文字以内、`num_predict=80`、`temperature=0.6`、`context=3072`。background: message 12 文字以内、rationale なし、`num_predict=40`、`temperature=0.35`、`context=1536`。

**Claude Code 分担** `persona-generator` subagent で 100 persona JSON、`runtime-refactor` で 1call/step 化、`prompt-ablation` で 4 条件比較、`video-overlay` で hall heat/despair heat/spotlight 実装。worktree isolation + hooks で `pytest + json validation + smoke render` を自動化。

**週次計画** 4/21 まで 15 agent・12 step stock demo、4/25 まで 24 agent pilot、4/29 まで `AT vs ART` の 2×2 ログ、5/2 まで図と主張の凍結（`hall_heat over time`, `generation別 dwell time`, `persona(e) transformation strip`）、5/3-6 で final export・動画・README、5/7 提出。35B は 5/3 以降の前景書き出しにだけ投入。

**リスクと縮退順序** Qwen3.5 35B の Ollama 回帰 → 即 `qwen3:30b` fixed に戻す。schema 破損 → soft schema + repair + retry で吸収、ダメなら background を action-only。wall clock → **先に step 数と background message 長を削る**、次に `background action-only`、最後に `foreground message-only`。video は推論後ログから再生成可能なので、最終週に rendering を推論と切り離す。

## 9. 判断軸（最適性の定義）

「最適」を以下の重みで判断してほしい:

1. **仮説の刺さりやすさ**（独自仮説と群像劇が動画1本で伝わるか）= 35%
2. **観察の豊かさ**（多ペルソナ × 3機種の差分と感情の多様性が見える）= 30%
3. **3 週間での実現可能性**（撤退ゲート全通過、Claude Code 実装前提）= 25%
4. **M4 Max 64GB スペックの活用度**（この環境だからできる規模）= 10%

迷ったら (1) > (3) > (2) > (4) の順で優先してほしい。
「安全だが平凡」より「攻めてるが通る」を選んでほしい。ユーザーは実体験者なので、
表層的なシミュレーションより**本質を刺す設計**を求めている。

ただし、**私は初心者であり、試行錯誤できる段階までの効率**も重視してほしい。
「攻めた最終形」と「そこに到達するための現実的なステップ」の両方が必要。
