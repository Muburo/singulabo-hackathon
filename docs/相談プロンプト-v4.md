## 3. 配布コードを使えなかった気づき（重要）

stock demo を動かしてユーザー（私）が直接見て、配布コードはスキャフォールドにならないと判断した。

| 項目 | 配布コード | 私の企画 |
|------|-----------|---------|
| エージェント | 男女（性別だけ） | 100 persona pool |
| 場所 | left_bar / right_bar | AT_bank / Hokuto_bank / ART_bank（複数台）|
| イベント | 火事（intensity, radius） | パチンコの抽選（当たり / ハマり / 連荘） |
| 観察値 | bar 占有率 | arousal / despair / 残金 / 連勝連敗 |
| LLM 入力 | 火事距離・bar 占有率 | persona numeric state + 台の状態 |

配布コードは **「人が場所を選ぶアニメーション」** で、私の **「台で打つ → 当たる/ハマる → 感情が動く物理ループ」** とは構造が違う。配布コードから流用できるのは「時間ループの骨格パターン」「LLM wrapper のパターン」「matplotlib での可視化パターン」程度で、本体は新規実装が必要。

## 4. 私がやりたい「物理実験」

### 4.1 観察したいもの

> **どういう属性の人が、どういう感情を持つか**

- ホールに複数の台（複数種類 × 複数台）
- ペルソナ 10〜100人
- 各ペルソナが時間とともにどう感情が動くかを観測

### 4.2 必須コンポーネント

| # | コンポーネント | 内容 |
|---|---|---|
| 1 | **台（Machine）の物理** | 当たり確率、連荘、ハマり、純増。台種ごとに分布が違う |
| 2 | **ペルソナの動的状態** | 残金、arousal、despair、連勝連敗、滞在時間 |
| 3 | **「打つ → 抽選 → 感情変化」のループ** | 各 step で台を引いて結果を反映 |
| 4 | **LLM の役割** | persona の行動決定（打つ/離れる/移動）+ 感情の自己申告 |
| 5 | **観察の出力** | 属性 × 時間 × 感情の log + 可視化 |

### 4.3 守りたい哲学（配布コードから継承）

- LLM には**数値のみ**渡す（「危険」「嬉しい」等の定性ラベルは禁止）
- persona は numeric state（年齢、依存重症度、stigma_barrier など）の組み合わせで定義
- 属性別の創発パターンを観察する

（例: 抽選結果は確率モデル、感情遷移は rule + LLM、行動は LLM）
